# 이벤트 회차(Occurrence) 기반 참여 기록·활성화 지표 — 상세 계획서

> **문서 목적**: 킹샷 게임 이벤트(정기·소규모·일정 가변)에 대해 멤버 참여 이력을 남기고, 조회·통계로 연맹/멤버 활성도를 지표화하기 위한 **데이터 모델·운영 규칙·구현 단계**를 한곳에 정리한다.  
> **범위**: 설계·계획만. 본 문서 작성 시점 기준 **코드/마이그레이션 변경은 하지 않는다.**

---

## 1. 배경과 목표

### 1.1 비즈니스 목표
- 각 **멤버**가 어떤 **이벤트**의 **어느 회차**에 참여했는지 이력을 남긴다.
- 위 데이터를 **조회**(필터·기간·연맹·등급 등)하고 **집계**하여 **활성화 정도**를 수치·비율로 본다.
- 운영자(동맹 관리자)가 **수동 입력** 또는 향후 **반자동/연동**으로 데이터를 유지할 수 있는 구조를 둔다.

### 1.2 성공 기준(검증 가능하게)
- 특정 기간 동안 **이벤트 타입별 참여율**(멤버 기준·연맹원 기준)을 산출할 수 있다.
- 동일 이벤트의 **서로 다른 회차**가 혼동되지 않고 구분된다(집계·리포트 기준 일치).
- 일정이 **연기·변경**되어도 “그 회차의 실제 기간”이 DB에 남고, 과거 집계가 깨지지 않는다(수정은 occurrence 단위).

### 1.3 도메인 용어(본 문서에서 고정)
| 용어 | 의미 |
|------|------|
| **이벤트 정의(Event definition)** | 게임 내 반복되는 이벤트 “종류”(이름·대략 주기·가중치 등). DB의 `events` 행에 대응. |
| **회차(Occurrence / Round)** | 그 이벤트가 **한 번 열린 구체적 기간**. 실제 `시작 시각`~`종료 시각`(또는 날짜 단위)으로 식별. |
| **참여(Participation)** | 특정 **멤버**가 특정 **회차**에 대해 참여 여부·점수·부가 정보를 가진 기록. |

핵심 원칙: **“2주마다 화요일” 같은 규칙은 집계의 진실이 아니다.** 진실은 **해당 회차에 확정된 실제 기간**이다.

## 2. 현재 구현 스냅샷(코드 기준, 계획의 출발점)

### 2.1 `events` (PostgreSQL `events`)
- `id`, `name`, `eval_weight`, `frequency`(enum: `daily` | `weekly` | `monthly` | `others`), `description`, `created_at`, `updated_at`
- 역할: 이벤트 **종류** 카탈로그에 가깝다.

### 2.2 `event_participations`
- `id`, `event_id` → `events.id`, `member_id` → `members.id`
- `is_participated`, `score`, `extra_info`(JSONB)
- `start_date`(NOT NULL), `end_date`(nullable)
- 유니크: `(event_id, member_id, start_date)`

### 2.3 현재 모델이 암묵적으로 하는 일
- **회차**를 별도 테이블로 두지 않고, `event_participations.start_date`(± `end_date`)로 **회차를 식별**하는 방식에 가깝다.
- 동일 `event_id` + 동일 `start_date`를 가진 여러 멤버 행 = 같은 회차로 해석 가능(운영 규칙에 의존).

---

## 3. 문제 정의 — 왜 “주기만”으로는 부족한가

### 3.1 게임 일정의 현실
- 정기 이벤트도 **정확히 N주 간격의 타임스탬프**로 고정되지 않는다.
  - 예: “2주에 한 번 돌아오는 **화·목**”처럼 **요일 앵커**가 있는 경우.
  - **연기·긴급 이슈**로 시작·종료가 밀리는 경우.
- 소규모 이벤트는 **며칠 단위**로 자주 바뀌고, 공지와 실제 종료가 어긋날 수 있다.

### 3.2 현재 스키마에서 생기는 운영 리스크
1. **회차 메타데이터 부재**: 공식 종료일, 연기 사유, “이번 회차 라벨” 등을 둘 **첫급 객체**가 없다(전부 참여 행에 흩어짐).
2. **참여 0명인 회차**: 아무도 기록하지 않으면 DB에 회차가 **존재하지 않음** → “이번 주기 열렸는지”만 추적하기 어렵다.
3. **`start_date` 불일치**: 운영자·도구에 따라 초 단위/타임존/오타가 다르면 **동일 회차가 여러 키로 쪼개져** 집계가 깨진다.
4. **`frequency` enum의 해상도**: `daily/weekly/monthly/others`만으로는 “격주 화목” 등을 표현하기 어렵고, **저장용 진실**과 **표시용 힌트**가 섞일 위험이 있다.

위 리스크는 “당장 쓸 수 없다”가 아니라, **데이터가 쌓일수록** 드러나는 유형이다.

## 4. 설계 방향 — 권장 개념 모델

### 4.1 3계층 구조(권장)
```
events (정의)
   └── event_occurrences (회차: 실제 기간)
            └── event_participations (멤버별 참여)
```

- **`events`**: 무엇인가(이름, 설명, 평가 가중치, **선택적** 주기 힌트·외부 slug 등).
- **`event_occurrences`**: 언제 열렸는가(**period_start**, **period_end**). 연기·변경은 **이 행 수정**으로 반영(과거 스냅샷이 필요하면 별도 이력 테이블은 후순위).
- **`event_participations`**: 누가 그 회차에 어떻게 참여했는가. FK는 **`occurrence_id` 우선**; `event_id`는 중복이지만 조회 편의를 위해 유지할지 팀에서 선택.

### 4.2 가변 일정을 DB에 넣는 규칙
1. 매 회차마다 **반드시 `event_occurrences` 한 행**(또는 현 단계에서는 **동일 의미의 단일 키**)이 존재해야 한다.
2. 집계·리포트는 **`period_start` / `period_end`** 를 기준으로 한다.
3. “이론상 다음 회차 예측”이 필요하면 **애플리케이션/캘린더 UI**에서만 힌트를 쓰고, **저장된 occurrence와 불일치하면 occurrence가 이김**.

### 4.3 ER 관계(텍스트)
- `events` 1 — N `event_occurrences`
- `event_occurrences` 1 — N `event_participations`
- `members` 1 — N `event_participations`
- (선택) `alliances`는 참여와 직접 FK하지 않고, 멤버의 소속·등급으로 조인해 집계

## 5. 목표 스키마 상세(제안 필드)

> 이름은 구현 시 팀 컨벤션에 맞게 조정 가능. 의미가 동일하면 된다.

### 5.1 `events` (기존 확장·정리안)
| 필드 | 설명 |
|------|------|
| 기존 유지 | `id`, `name`, `eval_weight`, `description`, 타임스탬프 |
| `frequency` | **표시·필터 힌트**로만 쓰거나, `others` + `description`/`metadata`로 이관 검토 |
| (선택) `slug` | 외부 사이트·엑셀·스크립트와 매핑 (`kingshot-data` 섹션명 등) |
| (선택) `kind` | `personal` / `alliance` / `mixed` 등 향후 집계 차원 |
| (선택) `schedule_hint` JSON | UI용: `{ "pattern": "biweekly", "weekdays": ["Tue","Thu"] }` — **집계에 사용 금지** 명시 |

### 5.2 `event_occurrences` (신규 — 권장)
| 필드 | 타입(개략) | 설명 |
|------|------------|------|
| `id` | PK | |
| `event_id` | FK → events | |
| `period_start` | timestamptz (권장) | 회차 시작(게임 서버 기준 또는 동맹 합의 기준 **하나로 통일**) |
| `period_end` | timestamptz nullable | 종료 미정이면 null 정책을 문서화 |
| `label` | string nullable | 운영자용 라벨, 예: `2026-W05`, `시즌3-2차` |
| `note` | text nullable | 연기·합병·공지 링크 등 |
| `created_at` / `updated_at` | | 감사·디버깅 |

**유니크 제약(선택)**: `(event_id, period_start)` — 동일 이벤트·동일 시작에 중복 occurrence 방지. (타임존 정책과 함께 결정)

### 5.3 `event_participations` (목표 형태)
| 필드 | 설명 |
|------|------|
| `occurrence_id` | FK → `event_occurrences` (**권장 필수**) |
| `member_id` | FK → `members` |
| `is_participated`, `score`, `extra_info` | 기존 유지 |
| `event_id` | 중복 컬럼: occurrence에서 유도 가능하나, 쿼리 단순화용으로 유지 시 occurrence 생성 시 동기화 규칙 필요 |

**유니크**: `(occurrence_id, member_id)` — 회차당 멤버 1행.

**기존 `start_date` / `end_date`**:  
- 이행 기간에는 occurrence의 기간과 **중복 저장**할지, participation에서는 제거할지 결정.  
- 권장: **참여 행에서는 제거**하고 occurrence만 진실로 둠(마이그레이션 시 기존 `start_date`로 occurrence 복원).

## 6. 현재 스키마만으로 “충분한가”에 대한 답

### 6.1 단기(데이터 소량·운영자 1~2명)
- **가능**: `event_id` + **회차 키로 통일된 `start_date`**(및 필요 시 `end_date`)만 엄격히 운영하면, 참여 이력·단순 집계는 가능하다.
- 반드시 문서화할 것:
  - 회차 키 = `start_date`의 **정규화 규칙**(예: 항상 KST 00:00, 또는 이벤트 시작일 00:00).
  - 동일 회차에 모든 멤버가 **동일한 `start_date`**를 쓸 것.

### 6.2 중장기(지표·대시보드·이력 정합성)
- **부족해지기 쉬움**: 위 3장 리스크대로, **`event_occurrences` 도입을 권장**한다.
- “전부 남기기”의 완성도는 **회차 엔티티**가 있을 때 올라간다.

## 7. 외부 참고 — 이벤트 종류 풀

- 커뮤니티 정리 사이트 예: [킹샷 이벤트 목록 (kingshot-data.com)](https://kingshot-data.com/ko/events/)  
  - 페이지 내 **섹션/이벤트 이름**은 `events.name` 또는 `slug` 매핑 후보로 삼을 수 있다.  
  - 해당 사이트는 비공식 팬 자료이므로, **공식 패치로 이름·주기가 바뀔 수 있음**을 전제로 “참고 카탈로그”로만 쓴다.

## 8. 집계·지표(요구사항 캘로그 — 구현 전 명세)

구현 시 SQL/뷰/API로 풀 항목을 아래에서 고른다.

### 8.1 멤버 단위
- 기간 `[T0,T1]` 동안 **참여한 회차 수** / **참여 가능 회차 수**(분모 정의 필요: “열린 occurrence 수” vs “우리가 추적한 이벤트만”).
- 이벤트 정의별 **참여율** = 참여 회차 수 / 해당 이벤트 occurrence 수(기간 내).
- `eval_weight` 가중 **가중 참여 점수**(간단 합·평균).

### 8.2 연맹 단위
- 연맹원 집합(또는 등급 R1~R5)별 **평균 참여율**, **미참여 인원 수**.
- 기간 추이(주간·회차별) 시계열 — 차트용 API shape 사전 정의.

### 8.3 운영 품질
- occurrence는 있는데 participation이 한 건도 없는 **미입력 회차** 목록(알림 후보).
- `extra_info` 스키마(버전 필드 등)를 나중에 정해 **임의 JSON 난립** 방지.

---

## 9. 이행 전략(단계별, 코드 작업은 별도 승인 후)

### Phase A — 문서·운영만 (스키마 유지)
- 회차 키 = `(event_id, start_date)` 규약을 **CONTEXT.md 또는 운영 가이드**에 고정.
- 관리자 UI에서 날짜 입력 시 **프리셋**(오늘 00:00 KST 등) 제공 여부 검토.

### Phase B — `event_occurrences` 추가 + 이중 쓰기(선택)
- 신규 데이터는 occurrence 생성 후 participation에 `occurrence_id` 연결.
- 기존 행은 배치로 occurrence 복원(같은 `event_id`+`start_date` 그룹 → occurrence 1개).

### Phase C — 스키마 정리
- `event_participations`에서 `start_date` 유니크 제거, `(occurrence_id, member_id)`로 전환.
- `start_date`/`end_date` 컬럼 제거 또는 읽기 전용 뷰.

### Phase D — 제품화
- 이벤트별 “회차 목록” 화면, “회차 일괄 생성(템플릿)” , 리포트 탭.

각 Phase는 **마이그레이션 롤백 계획**과 **백필 검증 쿼리**(행 수·NULL 수)를 체크리스트에 넣는다.

## 10. 리스크와 완화

| 리스크 | 완화 |
|--------|------|
| 타임존 혼선 | `timestamptz` + 저장 기준(예: KST) 문서화 |
| 운영자 입력 실수 | occurrence 단위 수정·감사 로그(후순위) |
| 이벤트 이름 변경 | `events` 수정과 과거 occurrence의 관계는 `event_id`로 유지 |
| 비공식 이벤트 목록과 불일치 | `slug`·`note`에 출처 URL 기록 |

## 11. 열린 결정 사항(승인 필요)

1. **회차 시각 정밀도**: 날짜만 vs 시각(게임 리셋 시각 반영 여부).
2. **분모 정의**: “활성화율”에서 분모를 occurrence 수로 할지, “해당 멤버가 속한 연맹이 참여 대상인 이벤트만”으로 필터할지.
3. **`frequency` 필드의 미래**: 유지 / 폐기 / 메타데이터 이전.
4. **참여 부정(미참여) 기록**: 미참여를 명시 행으로 둘지, absence는 집계 시 “없음=미참여”로 할지.
5. **`extra_info` JSON 스키마**: 최소 키 집합(예: `source: "manual"`, `entered_by`, `version`).

## 13. 요약

- **가변 일정**은 **회차 occurrence의 실제 기간**으로 저장하는 것이 정석이며, 규칙 기반 주기는 **힌트**로만 둔다.
- **현재 테이블만으로도** 엄격한 운영 규약 하에 단기 기록은 가능하나, **회차 테이블(`event_occurrences`) 추가**가 중장기 정합성·통계에 유리하다.
- 구현은 **Phase A→D**로 나누고, 마이그레이션·백필·UI를 순차 적용한다.
